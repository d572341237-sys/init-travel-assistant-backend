# 智能旅行助手前端需求文档

## 1. 项目定位

智能旅行助手前端用于承接后端 FastAPI + LangGraph 多 agent 旅行规划能力，为用户提供登录、自然语言旅行需求输入、候选景点选择、完整行程展示、天气建议、住宿地段建议和高德地图路线展示。

当前前端定位为 MVP 调试型产品界面，不做复杂商业化页面，不做营销落地页，优先保证核心交互链路完整、结果可读、地图路线可验证。

## 2. 目标用户

- 准备到某个城市旅游的普通用户。
- 希望用自然语言快速生成旅行计划的用户。
- 需要结合天气、景点地址、每日路线和住宿地段建议的用户。
- 项目展示对象：简历面试官、课程/项目评审人员、技术演示观众。

## 3. 核心目标

前端需要完成以下目标：

- 支持用户注册、登录、退出。
- 支持用户输入自然语言旅行需求。
- 支持两阶段旅行规划流程：先展示候选景点，再根据用户选择生成完整行程。
- 支持 SSE 流式接收 planning agent 输出。
- 将后端结构化 JSON 渲染为可读旅行计划。
- 根据旅行天数展示每日独立地图路线。
- 展示天气与穿衣建议、每日行程、住宿地段建议、其他提醒。
- 支持用户画像配置与后续个性化扩展。

## 4. 页面范围

### 4.1 登录/注册页

功能要求：

- 用户可输入用户名和密码。
- 支持登录。
- 支持注册。
- 登录成功后进入旅行规划页。
- 登录失败时展示友好错误提示。
- 未登录用户访问主页面时，应停留在登录态或跳转登录视图。

接口依赖：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 4.2 旅行规划主页面

页面应包含：

- 用户输入区。
- 发送按钮。
- 清空输出按钮。
- 退出登录按钮。
- 旅行规划结果区。
- 高德地图展示区。
- 当前登录状态提示。

输入区要求：

- 支持自然语言输入，例如：
  - `帮我规划汕头3天旅行，行程不要太紧凑，每天9点开始，18点结束，并结合天气给穿衣建议。`
  - `选择 1、2、3、4，安排3天，轻松一点`
- 发送中按钮应置灰，避免重复提交。
- 输入为空时展示错误提示。

### 4.3 用户画像页面或设置区

MVP 阶段可放在主页面的折叠区域或后续独立页面。

字段包括：

- 偏好旅行节奏：`relaxed`、`balanced`、`compact`
- 偏好每日开始时间。
- 偏好每日结束时间。
- 偏好城市。
- 偏好景点类型。
- 用户备注。

接口依赖：

- `GET /api/profile`
- `PUT /api/profile`

## 5. 核心业务流程

### 5.1 首次输入旅行需求

用户输入完整旅行需求后，前端调用：

- `POST /api/chat/stream`

后端总控 agent 识别城市、天数、行程节奏、时间范围等信息。

预期返回：

- 如果城市识别失败，返回追问。
- 如果是新的旅行需求，返回候选景点列表。
- 候选景点列表应提示用户可以输入编号或景点名称。

前端展示要求：

- 候选景点按编号列表展示。
- 保留后端原始文本中的选择说明。
- 不展示内部 agent 状态。
- 不展示 `thread_id`。

### 5.2 用户选择景点

用户输入类似：

- `选择 1、2、3、4`
- `我想去小公园、汕头老城、礐石风景区`
- `选择 1、2、3、4，安排3天，每天9点到18点`

前端继续调用：

- `POST /api/chat/stream`

后端流程：

- 天气 agent 查询天气。
- 景区地址信息 agent 查询景点地址。
- planning agent 生成最终结构化 JSON。
- 后端按最终每日行程调用高德驾车路径规划 API，补充真实路线。

前端展示要求：

- 流式输出阶段可显示正在生成的文本。
- 最终完成后应以结构化排版替换原始 JSON。
- 地图区域根据最终 JSON 渲染每日路线。

### 5.3 完整结果展示

最终结果应包含：

- 目的地。
- 行程天数。
- 行程节奏。
- 每日游玩时间。
- 天气与出行建议。
- 每日路线与游玩时间。
- 每个景点的地址、安排、交通提示和注意事项。
- 每日地图路线。
- 住宿地段建议。
- 其他建议。

## 6. 地图展示需求

地图展示使用高德 Web JS API。

配置接口：

- `GET /api/config/map`

地图展示规则：

- 按旅行天数生成地图卡片。
- 旅行 3 天应生成 3 张每日地图。
- 每张地图只展示当天景点和当天路线。
- 每张地图左上角或卡片标题必须明显显示第几天。
- 每张地图上方或下方必须展示路线摘要：
  - 第几天。
  - 起点 -> 终点。
  - 高德驾车约多久。
  - 距离多少。
- 如果当天只有一个景点，应展示景点位置，并提示当天无景点间驾车路线。
- 如果当天缺少坐标，应展示明确空状态，不应白屏。
- 地图渲染失败时应显示错误提示。

路线数据来源：

- `route_segments[].provider = "amap_driving"` 表示来自高德驾车路径规划。
- `route_segments[].polyline` 用于绘制路线折线。
- `route_segments[].duration_text` 用于展示驾车时长。
- `route_segments[].distance_text` 用于展示距离。

## 7. 数据展示格式

后端最终返回结构化 JSON，前端负责渲染。

关键字段：

```json
{
  "destination": "汕头",
  "days_count": 3,
  "pace": "relaxed",
  "daily_time_window": {
    "start": "09:00",
    "end": "18:00"
  },
  "weather_summary": {
    "summary": "天气概述",
    "clothing_advice": "穿衣建议",
    "umbrella_advice": "雨具建议",
    "safety_tips": []
  },
  "itinerary_days": [],
  "route_segments": [],
  "hotel_area_recommendation": {},
  "general_tips": []
}
```

前端不应直接把 JSON 原样展示给最终用户，除非解析失败。

## 8. 状态与错误处理

前端需要处理以下状态：

- 未登录。
- 登录中。
- 注册中。
- 等待输入。
- 请求发送中。
- SSE 连接中。
- planning agent 输出中。
- 完成输出。
- 后端错误。
- 地图配置缺失。
- 地图 SDK 加载失败。
- 地图路线数据缺失。
- 登录状态过期。

错误展示要求：

- 不展示 Python traceback。
- 不展示后端内部异常。
- 使用用户可理解的提示。
- 保留必要操作建议，例如重新登录、重新输入城市、稍后重试。

## 9. 接口清单

### 9.1 鉴权接口

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 9.2 用户画像接口

- `GET /api/profile`
- `PUT /api/profile`

### 9.3 旅行规划接口

- `POST /api/chat/stream`
- `POST /api/chat`
- `GET /api/plans`

主流程优先使用：

- `POST /api/chat/stream`

### 9.4 调试接口

- `GET /health`
- `GET /api/weather`
- `GET /api/attractions`
- `GET /api/route/context`
- `GET /api/config/map`

调试接口不一定需要暴露在正式前端主流程中。

## 10. SSE 事件处理

前端需要至少处理以下事件：

- `token`：追加流式文本。
- `message`：展示普通消息，例如候选景点列表或追问。
- `done`：最终结果完成，解析并渲染结构化结果。
- `error`：展示错误提示。

前端应忽略内部流程事件：

- `agent_selected`
- `agents_started`
- `node_start`
- `node_end`

## 11. 会话要求

- 前端不生成 `thread_id`。
- 前端不展示 `thread_id`。
- 会话由后端 HttpOnly Cookie 管理。
- 同一个浏览器会话内，用户选择景点时应能延续上一次候选景点上下文。
- 用户退出登录后，前端应清空当前输出和登录状态。

## 12. 视觉与交互要求

MVP 阶段视觉风格应简洁、工具型、可读性优先。

布局建议：

- 左侧输入区。
- 右侧结果区。
- 结果区上半部分显示旅行规划文本。
- 结果区下半部分显示每日地图。

响应式要求：

- 桌面端使用双栏布局。
- 移动端改为上下布局。
- 文本不应溢出按钮、卡片或地图说明区域。

## 13. 非功能需求

性能要求：

- 发送请求后立即进入 loading 状态。
- SSE token 到达后应立即更新输出。
- 地图 SDK 只加载一次。
- 地图失败不应影响文本行程展示。

安全要求：

- 不在前端保存 DeepSeek Key。
- 不在前端保存高德 Web 服务 API Key。
- 高德 Web JS API Key 只用于地图展示，并通过后端配置接口返回。
- 用户登录状态使用 HttpOnly Cookie。

兼容要求：

- 支持 Chrome、Edge 等现代浏览器。
- 支持 Windows 本地开发环境。

## 14. MVP 不做范围

当前阶段不做：

- 酒店实时库存查询。
- 酒店价格、评分、空房展示。
- 在线支付。
- 机票/火车票预订。
- 多人协同编辑行程。
- 拖拽调整路线。
- 收藏夹和社交分享。
- 商业化首页和营销落地页。

## 15. 验收标准

### 15.1 登录验收

- 用户可以注册新账号。
- 用户可以登录。
- 用户可以退出登录。
- 未登录时不能直接使用主流程。

### 15.2 旅行规划验收

- 输入城市和旅行天数后，系统返回候选景点。
- 用户选择景点后，系统返回完整旅行计划。
- 最终结果包含天气建议、每日行程、住宿地段建议。
- 如果城市识别失败，系统应追问用户明确城市。

### 15.3 地图验收

- 3 天旅行应展示 3 张每日地图。
- 每张地图显示第几天。
- 每张地图显示当天景点。
- 每张地图展示当天路线。
- 每条路线显示起点、终点、驾车时长和距离。
- 高德路线数据可用时，应展示“高德驾车约”。

### 15.4 异常验收

- 后端异常时前端显示友好错误。
- 地图加载失败时文本行程仍可正常查看。
- SSE 中断时前端可以恢复发送按钮状态。
