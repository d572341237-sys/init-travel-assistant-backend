## 2. 产品背景

用户在规划自由行时，通常需要同时查询目的地景点、天气、路线、每日时间安排和住宿区域。传统方式需要在多个 App 或网页之间切换，信息分散，规划成本高。

本项目希望通过大模型和多 agent 架构，把自然语言旅行需求转化为结构化行程规划，并结合高德地图能力展示真实驾车路线。前端承担用户交互、结果展示和地图可视化任务。

当前阶段重点不是做完整商业化旅游平台，而是完成一个面向简历项目展示的全栈 AI 旅行助手 MVP。

## 3. 产品目标

### 3.1 MVP 目标

- 用户可以注册、登录、退出。
- 用户可以输入自然语言旅行需求。
- 系统可以识别目的地、天数、节奏、每日时间范围。
- 系统可以先返回候选景点列表。
- 用户选择景点后，系统生成完整旅行规划。
- 最终结果包含天气建议、每日行程、住宿地段建议。
- 地图区域按旅行天数展示每日高德驾车路线。
- 用户画像可保存旅行偏好。
- 历史行程可查询。

### 3.2 项目展示目标

- 体现前后端完整交互。
- 体现 SSE 流式输出。
- 体现 LangGraph 多 agent 流程。
- 体现 Function Call / 外部 API 调用能力。
- 体现高德地图路线可视化。
- 体现数据库持久化和用户画像能力。

## 5. 目标用户

### 5.1 普通旅行用户

特征：

- 想快速规划旅行。
- 不想手动查多个平台。
- 对每天游玩时间、行程松紧程度有要求。
- 希望结合天气做准备。
- 希望看到地图路线。

核心诉求：

- 快速知道去哪里玩。
- 每天怎么安排。
- 天气如何，穿什么。
- 住在哪里比较方便。
- 每天路线是否合理。

### 5.2 项目评审或面试官

特征：

- 关注项目技术完整度。
- 关注 AI agent 架构是否真实落地。
- 关注前后端交互是否完整。
- 关注外部 API 是否真实接入。

核心诉求：

- 快速理解项目功能闭环。
- 看到 AI 规划、天气、地图路线、数据库等能力协同。
- 看到产品不是单纯大模型问答，而是结构化系统。

## 6. 核心用户故事

### 6.1 创建旅行计划

作为用户，我希望输入一句自然语言旅行需求，系统能理解我的目的地、旅行天数和偏好，并推荐候选景点。

示例：

```text
帮我规划汕头3天旅行，行程不要太紧凑，每天9点开始，18点结束，并结合天气给穿衣建议。
```

### 6.2 选择想去的景点

作为用户，我希望系统先列出目的地候选景点，让我选择真正想去的地方，而不是直接替我决定全部行程。

示例：

```text
选择 1、2、3、4、6、7
```

### 6.3 获取完整行程

作为用户，我希望在选择景点后，系统自动生成按天拆分的行程，并告诉我每天几点去哪里、怎么走、注意什么。

### 6.4 查看天气建议

作为用户，我希望系统结合目的地未来几天天气，告诉我是否需要带伞、穿什么衣服、有哪些安全提醒。

### 6.5 查看每日地图路线

作为用户，我希望按天查看地图路线，每天一张地图，能看到当天景点和高德驾车路线，而不是所有路线挤在一张地图里。

### 6.6 保存个人旅行偏好

作为用户，我希望保存我的旅行习惯，例如喜欢轻松行程、每天 10 点出发、偏好历史文化景点，后续规划时可以自动参考。

### 6.7 查看历史行程

作为用户，我希望能查看之前生成过的旅行计划，方便复用或回顾。

## 7. 产品整体流程

### 7.1 主流程

```text
用户登录
  -> 输入旅行需求
  -> 系统识别城市、天数、节奏、时间
  -> 返回候选景点
  -> 用户选择景点
  -> 天气 agent 查询天气
  -> 景区地址 agent 查询景点地址
  -> planning agent 生成结构化行程
  -> 后端调用高德驾车路径规划
  -> 前端展示完整行程和每日地图
  -> 保存历史行程
```

### 7.2 异常流程

城市识别失败：

```text
用户输入不明确
  -> 系统无法识别城市
  -> 前端展示追问
  -> 用户补充目的地
```

用户中途输入新目的地：

```text
已有候选景点上下文
  -> 用户输入新的城市需求
  -> 总控 agent 判断为新需求
  -> 清理上一轮候选景点
  -> 返回新城市候选景点
```

地图加载失败：

```text
文本行程正常展示
  -> 地图区域展示失败提示
  -> 用户仍可阅读行程
```

## 9. 功能模块需求

## 9.1 用户认证模块

### 功能说明

用户可以注册账号、登录系统、退出登录。登录状态由后端 HttpOnly Cookie 管理，前端不直接维护 token。

### 页面元素

- 用户名输入框。
- 密码输入框。
- 登录按钮。
- 注册按钮。
- 错误提示。
- 登录中状态。

### 业务规则

- 用户名不能为空。
- 密码不能为空。
- 登录失败时展示友好错误。
- 注册成功后自动进入登录态。
- 退出登录后清空当前页面输出。

### 接口依赖

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 验收标准

- 用户可以注册。
- 用户可以登录。
- 用户可以退出。
- 未登录时不能进入核心规划流程。
- 登录失败不展示后端 traceback。

## 9.2 旅行需求输入模块

### 功能说明

用户通过自然语言输入旅行需求。系统支持识别目的地、旅行天数、节奏、每日开始时间和结束时间。

### 页面元素

- 多行文本输入框。
- 示例文本。
- 发送按钮。
- 清空按钮。
- 当前请求状态。

### 输入示例

```text
帮我规划泉州2天旅行，轻松一点，每天9点开始18点结束。
```

```text
我想去汕头玩3天，喜欢老城和文化景点，行程不要太赶。
```

### 业务规则

- 输入为空时禁止发送。
- 请求中禁止重复发送。
- 发送后通过 SSE 接收结果。
- 前端不展示内部 agent 节点状态。

### 接口依赖

- `POST /api/chat/stream`

### 验收标准

- 用户可以发送自然语言需求。
- 请求中按钮置灰。
- 请求失败时按钮恢复。
- 返回候选景点或追问信息。

## 9.3 总控识别与追问模块

### 功能说明

前端需要展示总控 agent 的判断结果，但不展示内部推理细节。

### 业务规则

- 如果城市识别失败，展示追问。
- 如果用户输入新城市，按新旅行需求处理。
- 如果用户选择景点不明确，提示用户重新选择。
- 如果用户输入编号，视为选择候选景点。

### 前端展示

城市识别失败示例：

```text
我还没有识别出你想去的城市，请明确告诉我目的地。
```

选择景点提示示例：

```text
你可以回复编号，例如：选择 1、3、5。
```

### 验收标准

- 不明确输入时有追问。
- 新城市需求不会沿用旧城市候选景点。
- 不展示 `thread_id`。

## 9.4 候选景点模块

### 功能说明

用户首次输入旅行需求后，系统返回目的地候选景点。用户选择后再生成完整行程。

### 页面元素

- 候选景点列表。
- 景点编号。
- 景点名称。
- 景点类型。
- 地址。
- 选择说明。
- 当前规划偏好摘要。

### 业务规则

- 景点必须有编号。
- 用户可按编号选择。
- 用户可按景点名称选择。
- 用户可在选择时补充天数、节奏和时间。

### 接口依赖

- 主流程：`POST /api/chat/stream`
- 调试：`GET /api/attractions`

### 验收标准

- 用户输入城市后能看到候选景点。
- 用户能根据编号继续选择。
- 选择后进入完整行程生成流程。

## 9.5 完整行程展示模块

### 功能说明

用户选择景点后，系统返回完整结构化行程。前端负责将 JSON 渲染为可读内容。

### 页面元素

- 目的地。
- 行程天数。
- 行程节奏。
- 每日时间范围。
- 天气与出行建议。
- 每日行程列表。
- 住宿地段建议。
- 其他建议。

### 数据字段

```json
{
  "destination": "汕头",
  "days_count": 3,
  "pace": "relaxed",
  "daily_time_window": {
    "start": "09:00",
    "end": "18:00"
  },
  "weather_summary": {},
  "itinerary_days": [],
  "route_segments": [],
  "hotel_area_recommendation": {},
  "general_tips": []
}
```

### 业务规则

- 前端应优先解析 JSON。
- JSON 解析成功后展示格式化内容。
- JSON 解析失败时展示原始文本。
- 每日行程必须按天分区。
- 不直接向用户展示原始 JSON。

### 验收标准

- 天气建议展示正常。
- 每日行程展示正常。
- 住宿地段建议展示正常。
- 其他提醒展示正常。

## 9.6 天气建议模块

### 功能说明

展示天气 agent 查询结果和 planning agent 总结后的出行建议。

### 页面元素

- 天气概述。
- 穿衣建议。
- 雨具建议。
- 安全提示。

### 业务规则

- 天气信息放在完整结果靠前位置。
- 天气查询失败时，行程仍可展示。
- 天气失败提示应友好，不展示内部异常。

### 接口依赖

- 主流程：`POST /api/chat/stream`
- 调试：`GET /api/weather`

### 验收标准

- 用户能看到目的地天气建议。
- 用户能看到穿衣和雨具建议。

## 9.7 每日地图路线模块

### 功能说明

展示每日独立高德地图路线。后端会在最终 planning JSON 生成后，按每日相邻景点调用高德驾车路径规划 API，返回真实路线 polyline、距离和时长。

### 页面元素

- 地图区域标题。
- 每日地图卡片。
- 第 N 天标识。
- 当天路线概览。
- 地图画布。
- 景点 marker。
- 驾车路线 polyline。
- 路线说明列表。

### 路线说明字段

- 第几天。
- 起点。
- 终点。
- 高德驾车约多久。
- 距离多少。

### 数据字段

```json
{
  "day": 1,
  "origin": "小公园",
  "destination": "汕头老城",
  "origin_location": {
    "lng": 116.0,
    "lat": 23.0
  },
  "destination_location": {
    "lng": 116.1,
    "lat": 23.1
  },
  "distance_m": 1200,
  "duration_s": 600,
  "distance_text": "1.2 公里",
  "duration_text": "10 分钟",
  "polyline": [[116.0, 23.0], [116.1, 23.1]],
  "provider": "amap_driving"
}
```

### 业务规则

- 旅行几天展示几张地图。
- 每张地图只展示当天景点和当天路线。
- 地图必须显示第几天。
- 当天只有一个景点时，显示景点位置，并提示无景点间路线。
- 高德路线失败时，可以展示景点位置和错误提示。
- 地图失败不影响文字行程展示。

### 接口依赖

- `GET /api/config/map`
- 主流程返回的 `route_segments`

### 验收标准

- 3 天旅行展示 3 张地图。
- 每张地图显示第几天。
- 每张地图展示当天路线。
- 每条路线展示起点、终点、时长和距离。
- 高德路线数据可用时显示“高德驾车约”。

## 9.8 住宿地段推荐模块

## 10. 状态设计

## 10.1 页面状态

| 状态 | 说明 |
| --- | --- |
| 未登录 | 展示登录/注册入口 |
| 已登录未输入 | 展示输入框和引导文案 |
| 请求中 | 按钮置灰，展示 loading |
| 候选景点待选择 | 展示候选景点列表 |
| 行程生成中 | 展示 SSE 流式输出 |
| 行程完成 | 展示格式化行程和地图 |
| 请求失败 | 展示友好错误 |
| 地图失败 | 文本正常，地图显示失败提示 |

## 10.2 空状态

未输入：

```text
输入目的地、旅行天数和偏好后，我会先为你推荐候选景点。
```

无候选景点：

```text
暂未查询到可推荐景点，请换一个城市或补充更具体的地点。
```

无历史行程：

```text
暂无历史行程，完成一次规划后会自动保存。
```

地图无坐标：

```text
当前行程缺少可展示的地图坐标。
```

当天无路线：

```text
当天只有一个景点或缺少路线数据，暂无景点间驾车路线。
```

## 10.3 加载状态

| 场景 | 文案 |
| --- | --- |
| 登录中 | 登录中... |
| 注册中 | 注册中... |
| 连接后端 | 正在连接旅行助手... |
| 查询景点 | 正在查询候选景点... |
| 生成行程 | 正在生成旅行规划... |
| 加载地图 | 正在加载高德地图... |
| 生成路线 | 正在生成每日驾车路线... |

## 10.4 错误状态

| 错误 | 展示策略 |
| --- | --- |
| 登录失败 | 展示账号或密码错误 |
| 城市识别失败 | 追问用户明确目的地 |
| SSE 中断 | 提示重新发送 |
| 后端异常 | 展示通用友好错误 |
| 高德地图失败 | 地图区域展示错误，文本不受影响 |
| 高德路线失败 | 展示景点位置和路线缺失提示 |

## 11. SSE 事件处理

前端主要使用：

- `POST /api/chat/stream`

需要处理事件：

| 事件 | 处理方式 |
| --- | --- |
| `token` | 追加流式文本 |
| `message` | 展示普通消息或候选景点 |
| `done` | 解析最终结果并格式化展示 |
| `error` | 展示错误提示 |

需要忽略的内部事件：

- `agent_selected`
- `agents_started`
- `node_start`
- `node_end`

## 14. 会话与安全要求

- 前端不生成 `thread_id`。
- 前端不展示 `thread_id`。
- 登录态由后端 HttpOnly Cookie 管理。
- 前端不保存 DeepSeek API Key。
- 前端不保存高德 Web 服务 API Key。
- 高德 Web JS API Key 只用于地图 SDK 加载。
- 错误提示不展示后端 traceback。

## 15. 移动端适配

移动端要求：

- 页面改为单列布局。
- 输入区在上，结果区在下。
- 按钮允许换行。
- 地图高度降低到 240px 到 280px。
- 每日地图卡片纵向排列。
- 行程文本自动换行。
- 不出现横向滚动。
- 历史行程列表使用单列卡片。

## 16. 非功能需求

## 16.1 性能

- 地图 SDK 只加载一次。
- SSE token 到达后及时渲染。
- 地图加载失败不阻塞行程文本展示。
- 请求中避免重复提交。

## 16.2 可维护性

- 行程文本展示、地图展示、用户画像、历史行程应拆成独立模块。
- 地图渲染逻辑应只依赖最终 JSON，不直接依赖后端内部 agent 状态。
- 错误状态和空状态应统一封装。

## 16.3 可扩展性

后续可扩展：

- 行程重新生成。
- 单日行程调整。
- 景点删除。
- 历史行程收藏。
- 导出 PDF。
- 分享链接。
- 酒店真实查询。

## 17. 验收标准

## 17.1 登录验收

- 可以注册新用户。
- 可以登录已有用户。
- 可以退出登录。
- 登录失败有明确提示。
- 未登录不能使用主流程。

## 17.2 旅行规划验收

- 输入旅行需求后返回候选景点。
- 用户选择景点后生成完整行程。
- 城市识别失败时系统会追问。
- 用户输入新城市时不会沿用旧城市上下文。

## 17.3 行程展示验收

- 展示目的地、天数、节奏和时间范围。
- 展示天气和穿衣建议。
- 展示每日行程。
- 展示住宿地段建议。
- 展示其他建议。
- 不直接展示原始 JSON。

## 17.4 地图验收

- 旅行几天展示几张每日地图。
- 每张地图标明第几天。
- 每张地图展示当天景点 marker。
- 每张地图展示当天高德驾车路线。
- 每段路线展示起点、终点、驾车时长和距离。
- 地图失败时文本行程仍可查看。

## 17.5 用户画像验收

- 可以读取用户画像。
- 可以修改用户画像。
- 保存成功有提示。
- 保存失败有提示。

## 17.6 历史行程验收

- 规划完成后自动保存。
- 用户可以查看历史行程。
- 无历史行程时有空状态。

## 18. MVP 测试用例

### 用例 1：注册登录

步骤：

1. 打开页面。
2. 输入用户名和密码。
3. 点击注册。
4. 退出登录。
5. 使用同一账号登录。

预期：

- 注册成功。
- 登录成功。
- 状态栏显示当前用户。

### 用例 2：首次输入旅行需求

输入：

```text
帮我规划汕头3天旅行，行程不要太紧凑，每天9点开始，18点结束，并结合天气给穿衣建议。
```

预期：

- 返回汕头候选景点。
- 显示选择说明。

### 用例 3：选择景点生成行程

输入：

```text
选择 1、2、3、4、5、6
```

预期：

- 生成完整旅行规划。
- 展示天气建议。
- 展示 3 天行程。
- 展示 3 张每日地图。
- 地图路线显示高德驾车时长和距离。

### 用例 4：城市识别失败

输入：

```text
帮我规划一下旅行
```

预期：

- 系统追问目的地。

### 用例 5：中途切换城市

步骤：

1. 输入汕头旅行需求并得到候选景点。
2. 不选择景点，直接输入：

```text
我要去泉州旅游2天
```

预期：

- 系统识别为新需求。
- 返回泉州候选景点。
- 不生成汕头行程。

### 用例 6：地图失败降级

操作：

- 暂时移除或配置错误高德 JS API Key。

预期：

- 文本行程正常展示。
- 地图区域显示配置或加载失败提示。
